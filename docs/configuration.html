Configuration

Profiles, packages, dotfiles, and all the knobs you can turn.

Profiles

Profiles are INI files in the profiles/ directory that define what gets installed and configured. Each profile specifies packages, dotfile mappings, services, and setup scripts.

Listing and Inspecting Profiles

# List all available profiles
./setup.sh --list-profiles

# Show packages and details for a profile
./setup.sh --show-profile full

# Compare two profiles side by side
./setup.sh --diff-profiles minimal full

# Validate a profile's syntax and references
./setup.sh --validate-profile my-custom

INI Format Reference

All configuration files use INI format — no external parsers needed. The built-in parser supports sections, key-value pairs, inline comments, and multi-value keys.

# This is a comment
; This is also a comment

[section]
key = value
another_key = value1 value2 value3  # inline comment

[section.subsection]
nested_key = nested_value

Multi-value keys: Space-separated values on a single line are treated as a list (e.g., packages). You can also repeat a key on multiple lines to append values.

Anatomy of a Profile

A profile file has these sections:

# profiles/my-custom.conf

[profile]
name = my-custom
description = My custom setup
extends = minimal              # inherit from another profile (or false)

[packages]
shell = nushell
editors = neovim
cli_essential = ripgrep fd-find fzf
extra = my-favorite-tool       # add any tool your package manager knows

[dotfiles]
source = profiles/my-custom/   # directory within dotfiles/

[dotfiles.links.1]
src = .config/myapp/           # relative to dotfiles source
dest = ~/.config/myapp/        # absolute target path

[dotfiles.links.2]
src = .gitconfig
dest = ~/.gitconfig

[services]
enable = sshd docker           # services to enable on boot

[setup_scripts]
run = scripts/setup-ssh-agent.sh
run = scripts/setup-docker.sh  # scripts to run after setup

Profile Inheritance

The extends key lets a profile inherit all packages, dotfiles, and services from a parent. The child profile's values are merged on top.

[profile]
name = full
extends = minimal  # gets everything from minimal, then adds more

Set extends = false for a standalone profile with no parent.

Creating Custom Profiles

The quickest way to create a new profile:

# Scaffold from template
./setup.sh --create-profile my-server

# Edit the generated file
$EDITOR profiles/my-server.conf

# Validate it
./setup.sh --validate-profile my-server

# Run with it
./setup.sh --profile my-server

You can also copy an existing profile and modify it, or start from the example templates (profiles/gaming.conf.example, profiles/server.conf.example).

Package Definitions

Packages are defined in two places:

  • packages/common.conf — universal package names used across all platforms
  • packages/platforms/*.conf — platform-specific package name overrides

When a profile lists editors = neovim, the installer looks up the actual package name for your detected platform. For example, neovim might become app-editors/neovim on Gentoo or stay neovim on Ubuntu.

Adding Custom Packages

Add packages directly to your profile's [packages] section or create packages/custom.conf:

# packages/custom.conf
[packages]
extra = my-tool another-package

Then run:

./setup.sh --profile full --no-dotfiles

Dotfile Management

Dotfiles are stored in dotfiles/profiles/<profile>/ and linked to their target locations via symlinks.

dotfiles/
  profiles/
    minimal/
      .config/nushell/
      .config/nvim/
      .gitconfig
      .ssh/config
    full/
      shell/
      editors/
      multiplexer/
      cli/
      git/
      dev/

Each profile defines its own link mappings in [dotfiles.links.*] sections. Links are created idempotently — re-running setup won't break existing symlinks.

Conflict Detection

# Find broken symlinks and conflicts
./setup.sh --detect-conflicts

# Remove all managed symlinks
./setup.sh --unlink --profile full

Secrets Management

Secrets are encrypted at rest using git-crypt. Files matching patterns in dotfiles/.gitattributes are automatically encrypted:

  • secrets/** — all files in the secrets directory
  • **/.ssh/id_* — SSH private keys
  • **/*.gpg — GPG keys
  • **/api-tokens/** — API tokens
  • **/.env* — environment files
# Manage secrets
./setup.sh --secrets status
./setup.sh --secrets pull
./setup.sh --secrets push

# GPG key management
./setup.sh --gpg import key.asc
./setup.sh --gpg list
